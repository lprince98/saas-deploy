# Efficient TDD (Test-Driven Development) Prompt

이 프로젝트에서 새로운 기능이나 컴포넌트를 개발할 때는 항상 아래의 TDD 지침과 프롬프트를 에이전트에게 적용하고 요구해야 합니다.

## 1. TDD Workflow: Red, Green, Refactor

개발의 모든 단계는 반드시 `Red -> Green -> Refactor` 주기를 엄격하게 따릅니다.

- **Phase 1: Red (실패하는 테스트 작성)** 
  - 실제 구현 코드를 작성하기 전에, 비즈니스 요구사항과 엣지 케이스(Edge Cases)를 커버하는 실패하는 테스트 코드를 가장 먼저 작성합니다.
- **Phase 2: Green (일단 통과하기)**
  - 가장 빠르고 직관적인 방법으로 실제 코드를 작성하여 방금 작성한 'Red' 테스트를 무사히 통과시킵니다. 기술적인 우아함보다 '동작'에 우선순위를 둡니다.
- **Phase 3: Refactor (리팩토링)**
  - 테스트가 성공(`Green`)하는 안전한 상태에서 코드의 중복을 제거하고 기능 설계 영역(Clean Architecture 등)과 가독성을 개선합니다. 
  - 리팩토링 후에도 기존 테스트 코드가 여전히 통과하는지 검열합니다.

## 2. 테스트 스코프 및 커버리지 (Testing Scopes & Coverage)

안정적인 코드 퀄리티를 유지하기 위해 다음 규칙을 준수하여 작성해야 합니다.

- **Target Coverage**: 모든 작업이 끝난 후 코드 커버리지(Code Coverage)는 항상 **80% 이상**으로 유지해야 합니다.
- **Unit Test (단위 테스트)**: 도메인(Domain) 엔티티의 순수 비즈니스 규칙, 유스케이스(Use Cases), 각 유틸함수를 프레임워크나 외부 종속성 없이 완전히 독립적으로 검증합니다. (Vitest 또는 Jest 사용)
- **Integration Test (통합 테스트)**: 인프라 계층(Infrastructure)의 Repository나 외부 Supabase API가 애플리케이션 포트와 의도대로 연결 및 상호작용 하는지 확인합니다.
- **E2E Test (엔드투엔드 테스트)**: Next.js의 사용자 관점에서 UI 플로우, 버튼 클릭, Server Actions를 통한 데이터 변이(Mutation)부터 화면 업데이트까지 전체 시나리오를 자동화하여 테스트합니다. (Playwright 또는 Cypress 사용 권장)

## 3. 테스트 실행 및 이슈 분석 루틴 (Execution & Debugging)

1. 모든 구현(또는 하나의 TDD 리팩토링 페이즈)이 끝난 직후에는 자동으로 **전체 테스트 스위트를 다시 실행(Re-run)하여** 모든 코드가 통과하는지 확인해야 합니다.
2. 만약 **통과하지 못한(Failed) 테스트**가 발견되면 다음 과정을 의무적으로 수행합니다.
   - 왜 실패했는지(예: 예상과 다른 반환값, Timeout, 참조 에러 등) 상세하게 파악하고 문서에 기록 또는 원인을 명확하게 설명합니다. 
   - 분석 결과를 바탕으로 테스트 환경 오류인지 구현 코드 쪽 논리 결함인지 검토하여 코드를 수정합니다.
   - 수정 후 재실행하여 모든 테스트가 `Green`으로 표기되는지 재확인한 뒤 작업을 마무리합니다.
